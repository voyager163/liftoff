# Design: fix-seed-artifact-lifecycle

## Context

The manifest is the inventory of what Liftoff owns in a generated project; `validate` checks presence against it and `update` restores/upgrades from it. The seeded bootstrap change is the one generated artifact whose *correct* lifecycle ends in disappearance (archiving moves the folder), so inventory membership is wrong for it by construction. The emitted `migrate-to-liftoff` change already demonstrates the correct regime: written once, never tracked, archiving is a non-event.

## Goals / Non-Goals

**Goals:**

- Archiving the seeded change becomes a non-event for `validate`, `update`, and `doctor`.
- Projects generated by 0.2.0 heal without manual manifest surgery.
- The distinction (owned vs seed) is expressed in one place — the artifact category — not scattered special cases.

**Non-Goals:**

- No new manifest schema version — entries simply stop appearing; v2 readers are unaffected.
- No change to what gets written at create time (the seed change is still generated).
- No general "ignore list" configuration — the seed set is defined by templates, not by users.

## Decisions

### D1: `seed` is an artifact category, not a name list

The four bootstrap artifacts change category from `governance` to `seed`. `buildManifest` filters `category === 'seed'`; the reconciler skips render artifacts of that category (alongside the existing `manifest`/`liftoff-config` exclusions). Rationale: the category field already exists on every artifact, downstream code gets one predicate instead of four names, and future seed content (other starter material) inherits the behavior for free. Alternative considered: extend the `liftoff-config` special case list (rejected — that regime keeps entries in the manifest, which keeps `validate` failing after archive; seeds need to be off the inventory entirely).

### D2: Legacy healing lives in the manifest loader

`loadManifest` drops entries whose `logicalName` is in the legacy seed set (`openspec-seed-change-metadata`, `openspec-seed-proposal`, `openspec-seed-design`, `openspec-seed-tasks`) after the version check. Rationale: the loader is the single choke point — `validate`, `update`, and `doctor` all heal in the same release with no per-command logic, and the next `update --apply` rewrites the manifest without the entries, making the cleanup permanent. The filter is by name (not category) because legacy manifests record these entries with category `governance`.

### D3: The invariant this preserves

The manifest hash invariant ("contentHash records what Liftoff last wrote") is untouched — seed artifacts simply exit the recorded state. Reconcile behavior for every owned artifact is unchanged; the contract test's logicalName snapshot is unchanged because seeds are still rendered.

## Risks / Trade-offs

- [A user deletes the seed change instead of archiving it and expects update to restore it] → Accepted: seeds are one-shot by design; the message "run liftoff update to restore" was never promised for them. Documented in the scaffold README line if needed later.
- [Frozen manifest fixture in the contract test still contains seed entries] → Regenerate the fixture in this change; the loader filter also tolerates stale fixtures, so this is hygiene rather than correctness.
- [Future contributor adds seed content with an `owned` category by mistake] → The seed-lifecycle test archives the seed and asserts no drift; new seed content added under the wrong category shows up as a failing pattern quickly. Proportionate at this team size.

## Migration Plan

Ships as 0.2.1. No stored-format change: 0.2.0-generated manifests heal on read (D2); manifests written by 0.2.1 simply lack seed entries. Rollback is reverting the release — v2 manifests remain valid either way.

## Open Questions

None — the owned/seed split, loader-side healing, and category mechanism were settled during the feedback triage discussion.
